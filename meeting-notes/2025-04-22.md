3. Mappings in MIDS
   
   - MIDS uses SSSOM: <https://mapping-commons.github.io/sssom/> 
   
   - Mappings done for DwC and ABCD
   
   - Github repo: <https://github.com/tdwg/mids/tree/main/source/mappings> 
   
   - draft preview of the website that shows the mappings: <https://tdwg.github.io/mids/mappings/index.html> 
   
   - metadata file: <https://github.com/tdwg/mids/blob/main/source/mappings/sssom_dwc_biology_mappings.sssom.yml> 
   
   - actual mapping files: <https://github.com/tdwg/mids/blob/main/source/mappings/sssom_dwc_biology_mappings.sssom.tsv> 
   
   - Problematic cases: where one element maps to different external elements, where all of the target elements need to be present
   
   - Regex Removal: to remove values like “0”, “null”, “unknown”
   
   - Versioning of the mappings is tricky, needs to be addressed in this group
   
   - Not addressed yet are topics like: White space removal & absence of elements: in some cases it would be valid if an element is not present, but if it is present only certain values would be allowed.
   
   - It is possible to use custom fields that are not part of SSSOM, but it would be useful to align those within TDWG to ensure consistent usage
   
   - The goal of MIDS is not to have a standard to store data, but to calculate the MIDS score for data in different standards, which is why they are not called “terms” but “information elements”
   
   - In Latimer Core the mappings are not displayed on a separate page, but with the individual terms, whereas for MIDS it makes more sense to have them on a separate page.
   
   - The Biodiversity Data Quality (BDQ) Task Group has a vocabulary that specifies the terms Empty and NotEmpty, that could be reused here: <https://github.com/tdwg/bdq/blob/master/tg2/_review/docs/list/bdq/index.md#user-content-bdq_empty> 
   
   - BDQ Tests: <https://github.com/tdwg/bdq/blob/master/tg2/_review/docs/terms/bdqtest/index.md> , could be combined with the preprocessing parameters for MIDS
   
   - In a use case, the tests could be referenced using URIs and require to pass before the MIDS score is calculated
   
   - New tests could be defined to extend the BDQ, but they require an implementation and inputs/outputs.

4. General discussion on the direction of this group
   
   - We are currently discussing two different approaches: Use-case driven (looking at specific use cases and how they can be solved) vs theory-driven approach (more from a top down theoretical perspective for our requirements)
   
   - SSSOM is gotten a lot of traction since this group first met, should we skip the steps outlined in the Charter for Tasks 1 and 2 and just simply recommend it, or do we do a proper landscape analysis, look which ways standards are currently mapped and what other approaches are out there and evaluate their advantages or disadvantages? Our resources are limited, so focusing might be a useful approach but on the other hand, we might not want to say with confidence that we do recommend the best solution for the different requirements for the TDWG standards. 
   
   - We also shouldn’t overcomplicate things while keeping the tricky use cases in mind.
   
   - Consensus for moving forward: A compromise between the two approaches: we will continue with Tasks 1 and 2, but with a clear focus on SSSOM, but at the same time, we will also look for cases where it doesn’t work or other reasons not to just flat out recommend it. This should be included in the survey. If we find special cases not covered, we might extend its usage to our needs, while also getting in contact with the SSSOM community, to let them know about the issues we encountered, and to ask for help/recommendations and to maybe even steer the development of the standard to our needs.
   
   - New mappings are necessary in the future anyways, with the new Darwin Core Data Package, so we should be able to provide a preliminary recommendation for the other standards to map to DwC DP.  Maybe even use the mappings from DwC to DwC DP to transfer standards automatically, even though close matches might cause issues.
   
   - Should there be a mapping repository? Probably not, to not keep multiple places for the mappings, but a registry instead. Hanna Koivula presented one concept in Hobart (<https://doi.org/10.3897/biss.7.112223> ). Maybe this is a useful tool for the future. We should check back, how this has developed. SSSOM also has a list of mappings.
